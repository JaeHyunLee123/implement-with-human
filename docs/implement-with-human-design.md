# implement-with-human 설계 결정

`implement-with-human`은 기존 `implement`와 같이 스펙 또는 티켓을 구현하되, 인간이 검토 가능한 작은 변경만 한 번에 진행하고 각 변경 뒤 인간의 명시적 승인을 기다리는 사용자 호출 전용 스킬이다.

## 구현 시작

1. 에이전트는 요구사항과 코드베이스를 탐색하고, Review Slice의 순서·검증 방법·예상 변경량을 담은 Slice Plan을 제시한다.
2. 인간이 Slice Plan을 승인한 뒤에만 첫 슬라이스를 구현한다.
3. 작업 트리에 기존 변경이 있으면 에이전트는 이를 보고하고, 현재 작업에 포함되는지 인간 확인을 받은 뒤 진행한다.
4. 진행 상태를 남길 Progress Checklist가 필요하다. 에이전트는 임의로 만들지 않고 위치를 먼저 묻는다. 기존 요구사항 또는 스펙 문서가 있으면 그 문서에 추가하도록 추천하고, 없으면 저장소 루트나 `docs/` 아래의 새 파일을 추천한다.

## Review Slice

- 하나의 독립적으로 검증 가능하고 커밋 가능한 변경 단위다.
- 테스트와 수동 문서를 포함하여 `git diff --numstat`의 추가·삭제 줄 합계가 300줄 이하여야 한다. 생성물과 lockfile은 한도에서는 제외하지만 리뷰 자료에 별도로 공개한다.
- 리뷰 피드백으로 생긴 후속 커밋도 최초 슬라이스 커밋의 부모부터 계산한 누적 diff에 포함한다.
- 분리할 수 없어 300줄을 넘기는 경우, 구현 전에 이유·예상 줄 수·리뷰 전략을 제시해 인간의 명시적 예외 승인을 받는다.
- 동작 변경과 무관한 리팩터링 또는 포맷 변경은 별도 슬라이스로 계획한다. 새 동작에 필요한 최소 구조 변경은 같은 슬라이스에 둘 수 있지만 리뷰 자료에서 구분한다.
- 커밋 메시지는 슬라이스 번호가 아니라 저장소의 기존 관례와 변경 내용을 따른다. 진행 위치는 Progress Checklist가 관리한다.

## 구현과 검증

- 설치되어 있으면 `/tdd` 스킬을 사용한다. Slice Plan에는 테스트 seam과 검증 방법을 기록하며, TDD가 적합하지 않다면 그 이유를 계획 단계에서 설명한다.
- 각 슬라이스는 관련 테스트와 타입체크 또는 린트를 통과해야 한다.
- 마지막 슬라이스와 공유 기반 코드·보안·마이그레이션처럼 영향이 넓은 슬라이스는 전체 테스트 스위트도 실행한다.
- 각 슬라이스를 커밋한 뒤 그 커밋의 부모를 기준점으로 `/code-review`를 실행한다. 명백하고 슬라이스 범위 안인 지적은 에이전트가 수정·재검증·재리뷰한다. 해석 또는 범위 결정을 요구하는 지적은 인간에게 제시한다.

## Review Gate

1. 에이전트는 커밋 후 Review Package를 제시하고 `awaiting-review` 상태로 멈춘다.
2. `승인`, `approve`, `다음 슬라이스 진행`처럼 명시적인 승인만 다음 슬라이스를 열 수 있다. 질문·의견·비공식적인 긍정 반응은 승인으로 간주하지 않는다.
3. 인간 피드백은 현재 슬라이스에서만 처리한다. 수정·검증·재리뷰 뒤 다시 Review Gate에서 승인을 기다린다.
4. 계획의 범위·순서·슬라이스 경계를 바꿔야 하면 Plan Revision을 제시하고 인간 승인을 받는다. 승인된 슬라이스 내부의 작은 구현 선택은 에이전트가 결정한다.

## Progress Checklist

각 슬라이스는 범위, 검증 명령과 결과, 커밋 SHA를 기록하고 다음 상태 중 하나를 가진다.

- `planned`
- `implementing`
- `awaiting-review`
- `approved`
- `deferred`
- `dropped`

`deferred`와 `dropped`는 인간의 명시적 지시로만 설정한다. 에이전트는 슬라이스를 임의로 건너뛰지 않는다.

## 완료

모든 계획 슬라이스가 승인된 뒤, 전체 테스트 스위트와 전체 변경 범위에 대한 `/code-review`를 실행한다. 에이전트는 최종 통합 Review Package를 제시하고 인간의 명시적 승인을 받은 뒤에만 구현을 완료로 처리한다.
